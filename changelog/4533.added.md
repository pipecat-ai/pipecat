- Added a `realtime_service_mode: bool = False` kwarg on `LLMContextAggregatorPair`, opting the pair into realtime (speech-to-speech) LLM behavior. When set, user messages are written to context when the assistant response starts rather than on user-turn-end frames — so context stays correct even when the realtime service emits no turn frames at all — and turn-end strategies stop waiting for transcripts before signalling end-of-turn, keeping transcript latency off the critical path in local-VAD-driven realtime pipelines. Cascade (non-realtime) behavior is unchanged when the kwarg is omitted. Note: when the assistant response starts before the user's transcript has arrived (typical for OpenAI Realtime, Azure Realtime, Grok, Inworld — they deliver the finalized user transcript shortly after starting their response), the user-message write is deferred by one `ttfs_p99_latency` window (captured from `STTMetadataFrame`, with a `DEFAULT_TTFS_P99` fallback) to let the transcript land — mirroring the cascade-mode wait after `UserStoppedSpeakingFrame`. A failsafe flush runs at assistant-response-end so the user message always precedes the assistant message in context.
